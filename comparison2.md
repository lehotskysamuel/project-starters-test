# Post-init Starter Comparison: `router/` + `shadcn init` vs `shadcn/` vs `monorepo/`

This document compares the three React-Router-based starters after `router/` has been mutated in place by running `shadcn init` on top of it. See `comparison.md` for the wider comparison across all six starters and for the target stack definition.

## Context

Per `todo-v2.txt`, the working hypothesis was that `router/` + `shadcn init` would become our base. This document evaluates whether that holds now that the init has actually been executed against `router/` in place.

Command that was run inside `router/`:

```bash
bunx --bun shadcn@latest init --base radix --preset nova --no-monorepo --no-rtl --no-reinstall -y --force
bun install
```

For reference, `shadcn/` and `monorepo/` were originally generated with:

```bash
# shadcn/
bunx --bun shadcn@latest init --template react-router --name shadcn --base radix --no-monorepo --no-rtl -y

# monorepo/
bunx --bun shadcn@latest init --template react-router --name monorepo --base radix --monorepo --no-rtl -y
```

So `router/` (post-init) and `shadcn/` were generated by the same shadcn CLI choosing the same preset (`nova`) / base (`radix`), but with a different *starting point*: `router/` already had a React Router project on disk (`create-react-router` default template), whereas `shadcn/` was built from `shadcn init --template react-router` on an empty directory.

## Conflicts left behind by `shadcn init` on `router/`

This is the important section. `shadcn init` did not cleanly overwrite `router/`'s existing RR template — it *added* on top of it. The result is a file set that has no analogue in either `shadcn/` or `monorepo/`:

1. **`app/app.css` has two competing theme blocks.** The original RR block is still there:

   ```css
   @theme {
     --font-sans: "Inter", ui-sans-serif, system-ui, sans-serif, ...;
   }

   html, body {
     @apply bg-white dark:bg-gray-950;
     @media (prefers-color-scheme: dark) { color-scheme: dark; }
   }
   ```

   Immediately followed by the shadcn `@theme inline` block declaring `--font-sans: 'Geist Variable', sans-serif;` and the `:root` + `.dark` OKLCH tokens, plus a `@layer base { body { @apply bg-background text-foreground; } }`. The later block wins for `--font-sans` and for `body` background (Tailwind `@layer base` beats element-level `@apply` with the same specificity because it's declared after), but the old block is still dead weight and the Inter `@theme` entry is unreachable. The `html, body { @apply bg-white dark:bg-gray-950 }` rule technically still applies whenever the `.dark` class isn't on an ancestor, causing a `bg-white` flash that shouldn't exist.

2. **`app/root.tsx` still exports `links()` with the Google Fonts preconnect + Inter stylesheet** — but `@fontsource-variable/geist` was added to `package.json` and is being imported by `app.css`. So two fonts are being loaded and only Geist is wired into `--font-sans`. The Inter fetch is pure waste.

3. **`app/welcome/welcome.tsx` + `app/welcome/logo-{dark,light}.svg` are still present**, and `app/routes/home.tsx` still reads `import { Welcome } from "../welcome/welcome"`. `shadcn/` deleted all of this and replaced `home.tsx` with a button demo; `router/` (post-init) did not.

4. **`tsconfig.json` has no `baseUrl`.** `shadcn/` and `monorepo/apps/web` both add `"baseUrl": "."`. Without it, `paths: { "~/*": ["./app/*"] }` is technically still resolved (TS allows `paths` without `baseUrl` since 5.0 and infers `baseUrl` as the directory of the tsconfig), but this differs from how `shadcn/` ships.

5. **`vite.config.ts` uses Vite 8's new `resolve.tsconfigPaths: true` option** instead of the `vite-tsconfig-paths` plugin that `shadcn/` and `monorepo/` ship. Functionally equivalent; structurally different. Vite 8 is newer than anything `shadcn/` or `monorepo/` use (both on Vite 7).

6. **No Prettier was installed.** `router/` had no Prettier before and `shadcn init` does not add it. `shadcn/` ships `.prettierrc`, `.prettierignore`, `prettier`, `prettier-plugin-tailwindcss`, and a `format` script. `router/` (post-init) has none of that.

7. **No shadcn components were added** — only `lib/utils.ts` (`cn`). `shadcn/` and `monorepo/` each ship `button.tsx` out of the box. This is actually a clean improvement (fewer files to delete) but worth noting.

8. **Docker.** `router/`'s `Dockerfile` (node:20-alpine multistage, from `create-react-router`) was left untouched. `shadcn/`'s `Dockerfile` is the shadcn-CLI-generated variant. Both exist, but they're not identical.

## Verdict: does the v2 hypothesis hold?

The hypothesis from `todo-v2.txt` was: `router/` + `shadcn init` is our base. Three problems with the output as-is:

1. **Dead code from the RR welcome template is still in `router/`.** `app/welcome/`, the `Welcome` import in `routes/home.tsx`, the Google Fonts `links()` in `root.tsx`, and the old `@theme { --font-sans: Inter }` + `html, body { bg-white }` block in `app.css` are all leftover. `shadcn/` was generated from a clean slate so it never had them. Cleaning them up is mechanical but non-zero work: delete `app/welcome/`, rewrite `routes/home.tsx`, strip the `links()` export from `root.tsx`, and remove the duplicate block from `app.css`.

2. **Prettier is not installed.** `shadcn/` ships Prettier + `prettier-plugin-tailwindcss` with `tailwindFunctions: ["cn","cva"]` and a `format` script. `router/` (post-init) has none of this. Copying all four Prettier assets from `shadcn/` is the minimum.

3. **No preseeded `button` component.** `shadcn/` ships `app/components/ui/button.tsx`; `router/` (post-init) does not. Easy fix (`bunx --bun shadcn@latest add button`) but it's a deliberate gap.

**Everything else is a win for `router/` (post-init):**

- React Router 7.14.0 (newest of all three — `shadcn/` is on 7.13.1, `monorepo` on 7.12.0).
- Vite 8 (newer than the Vite 7 in `shadcn/` and `monorepo`).
- Vite 8's `resolve.tsconfigPaths: true` replaces the `vite-tsconfig-paths` plugin dependency.
- Multi-stage Dockerfile from `create-react-router` is preserved.

**Versus `monorepo/`:** `monorepo/` still has the same drawbacks as before (Turbo overhead, per-package `tsconfig.json`s, cross-package CSS `@source` globs, older RR 7.12.0). Its unique wins — `zod`, `@remixicon/react`, `packageManager`/`engines` pins — remain one-line additions. The monorepo is only worth it if a second app is coming.

**Bottom line:** `router/` + `shadcn init` is now the base with the best raw React Router + Vite versions. It's not *drop-in equivalent* to `shadcn/` — it still needs ≈5 files of cleanup (welcome page, `root.tsx`, `app.css`, `routes/home.tsx`, Prettier configs, `button` component) — but those are trivial, reversible changes, and the result is a superset that `shadcn/` can't match on versions.

## Remaining work on top of `router/` (post-init)

**To reach parity with `shadcn/`:**

1. Delete `app/welcome/` directory.
2. Strip the `links()` export (Google Fonts Inter preconnect + stylesheet) from `app/root.tsx`.
3. Remove the duplicate top block (`@theme { --font-sans: Inter ... }` + `html, body { bg-white ... }`) from `app/app.css`.
4. Replace `app/routes/home.tsx` with the button demo (or a minimal landing page).
5. Copy `.prettierrc`, `.prettierignore`, and add `prettier` + `prettier-plugin-tailwindcss` to devDeps, plus a `format` script.
6. `bunx --bun shadcn@latest add button` (matches what `shadcn/` ships preseeded).

**To also pick up the `monorepo/`-only wins:**

7. `bun add zod`.
8. `bun add @remixicon/react` (optional — `lucide-react` alone is fine for our stack).
9. Add `"packageManager": "bun@1.3.11"` and `"engines": { "node": ">=20" }` to `package.json`.

**To satisfy the rest of the target stack (shared with `comparison.md`):**

10. React Compiler (`babel-plugin-react-compiler` + RR Vite plugin Babel options).
11. Playwright (`bun add -D @playwright/test`, `bunx playwright install`, `e2e/`, `playwright.config.ts`).
12. SSG: set `prerender: true` (or an explicit route array) in `react-router.config.ts`.
13. `bun add zustand @tanstack/react-query lodash axios`.
14. ESLint — copy the flat config from `vite/` (or `orange/`).
15. Vitest, MSW, Testing Library, Storybook, tRPC — add when needed.
